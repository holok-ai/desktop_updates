The desktop application must handle 4 types of files: files attached by the user to prompts provided to the LLM, files attached to context by tool functions, files generated by a tool function and a file generated as part of the response from an LLM. 

An LLM can perform tool functions like list_folder, read_file and write_file that require a file path. In the desktop, file paths are virtual file paths. This prevents the user prompts and the LLM from having access to any physical or system paths, mounts or resources. 
In shared projects, virtual paths allow folders and files to be shared by project members. For consistency, virtual paths are used in general threads, personal projects and shared projects.

A virtual path, for a folder or as the path for a file, is used when providing file locations to the LLM. This results in virtual paths being stored in the request and responses. When the LLM makes a call to a tool function, the Desktop app can inspect the virtual path, validate that it is valid (and not attempting to access a location outside the allowed list), generates a physical path for the virtual one and allows the local operation to continue with the physical path. 

The UI will display file thumbnails or links using appropriate UI elements that display a file attached to a prompt, or one generated from a response. The UI will work solely on a file id. 

File data used in the UI and the tool functions will include a unique id (to locate the file in the File Storage Service), folder path, file name and file type and metadata like create and last update dates.  Virtual paths allow users to run the Desktop on any computer and have files used or generated in stored prompts available to them. 

For general threads, the folder path is a virtual path that has the text "/user" as its root (where "user" is system managed to be the current user). An example of a virtual path in a general thread is "/user/finance/2025/Q2". Because the current user owns all their general threads, there is only one "/user" root for all general threads. This makes all files shareable across all threads, which is identical to a physical file system. 

For personal and shared projects, the file path is a virtual path that starts with "project" such as "/project/development/pr_review". The root of "project" represents a virtual path where all project members can add, rename or delete folders, as well as files. 

For general threads and projects, either a file id or a virtual file path and file name can be used to access file contents, virtual folder and metadata. A file id can be used to read the associated virtual data for a file. The UI, IPC handlers and tool functions can use either a fileid or a virtual path to access a folder or a file. 

To support being able to display a file using a file id, the Desktop application will need a File Storage Service. This will provide the ability to upload/save a physical file to a virtual folder and get a unique file id. For example, if the user uploads a Word document in a general thread, the UI would use the File Storage Service to save the physical path, file name and file contents, and in exchange receive a virtual file path and a file id. The desktop will use a File Id internally to efficiently pass, store and retrieve a file and have access to its contents. If a prompt is being sent to an LLM and contains a file id, the desktop will replace each file id with the actual file contents (for a file) and a virtual folder path if it's a folder. When the LLM response includes a file (such as an image generated by Gemini imagen), the file data is saved to the virtual file system using the File Storage System and the contents are replaced with a fileid tag from the File Storage System. The UI controls that display files, images, audio, movies and other media will get the actual content using the file id.

The File Storage Service will provide folder and file operations, compression, security at rest and data integrity. It will be a typescript API front end that uses either a cloud-provider such as AWS S3 or Azure File Storage, or an on-premise service such as seaweed FS or garage. The File Storage Service will verify the authentication of the current user (ie is user logged in and have a valid token), as well as for shared projects, verify that create, modify or write a file/folder has project editor or owner role. It will also ensure concurrent access to the virtual file system. (Using seaweed or garage, it has built in concurrent user access support, soft deletes, retention policies, as well as compression.) It will use the existing Desktop logger to log folder and file operations. Errors that prevent the reading or saving of a file by the UI will be displayed with a toast message. All file operations, successful and failed, will be written to the Desktop application log. 

File handling has these constraints and capacities:
- an individual file may not be larger than 15MBytes
- a user may not have more than 1000 files for general threads
- each project may not have more than 100 files (or more than 100 * 15 MByte total)
- file types are allowed except those in this blacklist exe, dmg, msi, ps1, bat, sh, cmd, vbs, jar, app

When cloud storage or the File Storage System is not reachable, the Desktop will allow the user to submit prompts but will not support attaching files. Likewise if the Desktop application receives files while the File Storage System is offline. It can display them but will not store them such that the next time the message with a file is displayed, the file will not be displayed. The Desktop application  should check the status of the File Storage System on start up, and consider it online if the health endpoint and a folder list and file read is successful. 

Command shell processing will require additional analysis and development to use virtual folders. Additionally, there are security issues with allowing access to a user's actual operating system. Command shell tool functions (such as windows_shell and unix_shell) should be given access to virtual folders and not physical ones. However, the mechanism to replace the physical devices and folders in the shell with virtual ones is not clear. Initially, we will let the command shell tool functions access the allowed folders. 

One promising option is to use Go and the mvdan library which implements a mini Unix shell, command preprocessor, a virtual file system, an approved command list and custom logic for executing each supported command. Go can be compiled on Windows, MacOS and Unix.  mvdan/sh is a pure Go implementation of a POSIX shell parser and interpreter that enables complete shell functionality without external dependencies or system shell access. The library consists of three key components: syntax for parsing shell scripts into an AST, interp for executing parsed programs, and expand for handling variable and glob expansion. Custom commands are defined by providing an ExecHandler function to the interpreter, which intercepts command execution and allows you to implement built-in commands like ls, cat, grep, and sed in Go code. Virtual filesystem support is achieved through pluggable handlers (OpenHandler, ReadDirHandler, StatHandler) that intercept all file operations, allowing commands to operate on an in-memory filesystem with no access to the host system. This makes mvdan/sh ideal for sandboxed shell execution where security requires complete isolation from the real filesystem.
